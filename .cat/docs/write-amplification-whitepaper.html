Write Amplification in a Persistent KV-Cache Warm Loop

A background cache warmer was rewriting its entire prefix to disk on every pause, producing write volume quadratic in the number of resumes. This reports the diagnosis, an exact architectural cost model derived before measurement, the fix, and — at equal length — the claims the measurements forced us to withdraw.

01The symptom

The package keeps a persistent prefix KV-cache so a long document is prefilled once and reused across many prompts. A background warmer builds that prefix incrementally, yielding to interactive work between chunks and resuming afterwards.

Each resume was writing a fresh whole-prefix snapshot and evicting the previous one. Warming one 183,018-token log across five observed resumes wrote 7.1 GB to reach 51% of the file, projecting to roughly 34 GB for a single document — plus a full read of the previous snapshot on every resume.

02An exact cost model

Snapshot size is affine in prefix length: bytes(N) = A·N + M. The useful result is that both constants are architectural — computable from a model's config.json before any weights are loaded:

  • A = attention layers × 2 (K,V) × kv-heads × head-dim × 2 bytes — the per-token attention cost.
  • M = linear layers × (value-heads × key-head-dim × value-head-dim × 4 bytes) — the fixed recurrent state, plus a small convolution term.

Predicting first and measuring second matters: fitting A and M to file sizes and then checking them against those same file sizes is an arithmetic identity, not a measurement. Predicted against measured:

The 35B's residual — 64.4 MB measured against a 62.9 MB state-space-only prediction — is 3.2 MB, or roughly 107 kB per linear layer, which is the convolution state the prediction deliberately omits. The 1.7B's 5.5 kB is the safetensors header.

This decomposition is what makes the rest tractable. The 35B is 20 KiB per token of sliceable attention KV, plus 64 MB of recurrent state that does not grow at all. The two halves have entirely different economics and must be costed separately.

03Root cause

Three mechanisms compose into the quadratic:

  • The save path copies the whole live cache and serialises it unconditionally. There is no delta or append path for any model type.
  • The catalog's record planner gates on at least one block being new, then re-plans every boundary — so a snapshot covering N blocks claims all N, not the handful that are new.
  • Commit therefore transfers every boundary to the new file, the previous snapshot is left owning none, and it is collected as an orphan. This is the evicted 1 observed on every round: not LRU pressure, but correct orphan collection of a genuinely redundant file.

An earlier reading of this code held that the planner already skipped catalogued blocks, and that only the storage layer was non-incremental. That was wrong in a way that matters: the index is block-granular, but the write plan is not, and a source comment in the coordinator asserts the opposite.

04The intervention

The warm entry point returns only an outcome enum, so the live cache it builds is a local and is dropped on return. A caller cannot hold it. Every resume must therefore begin by reloading the entire prefix from disk, and must end by writing it back.

The fix is to hold the live cache across pauses, keyed by an identifier, and persist on completion or abandonment rather than on every yield — the pattern the package already ships for multi-turn conversations. Persistence becomes a durability decision instead of the resume mechanism.

A container constraint shapes the API: the model container's perform is generic over a Sendable return, so a value containing the cache cannot be returned out of it at all. Returning an opaque handle to the caller is therefore not merely inadvisable — it does not compile without an escape hatch the codebase has a standing check to keep removed. The holder must live library-side and be reached through a Sendable key.

05Method

The intervention was measured before being implemented. A benchmark harness drives the counterfactual arm entirely in harness code against the existing public API, so the hypothesis could be falsified without committing to a design.

Five design decisions, each answering a specific way the experiment could have lied:

  • Predicted, not fitted. Constants derive from config before weights load, so the comparison is falsifiable.
  • File sizes are labelled "logical" and are never the headline. Summed file sizes are not I/O. Wall clock, forced device flush, and per-process disk counters are the primary observables.
  • Both arms are pinned to an identical chunk sequence so the losslessness check tests residency rather than floating-point associativity.
  • Every arm asserts it did work. Without a per-chunk offset assertion, "performed no I/O" is indistinguishable from "performed nothing", and both residency gates would pass on an empty function body.
  • A positive control. A losslessness gate that cannot fail proves nothing, so the comparator must first be shown to detect a known difference.

A further hazard, orthogonal to this work but worth recording: the generation API does not run inside the caller's perform — it boxes the cache and launches a detached task, and its stream is Sendable, so the compiler permits returning it and draining it outside the serialised domain. That places two live caches on the GPU concurrently with no diagnostic. The rule is a convention today; it should be documented.

Correctness

A cache held across seven pause/resume cycles is bit-identical to one prefilled in a single pass — zero maximum absolute difference across every attention and recurrent tensor, on all three models. Generating directly from the held cache, with no persist and no reload, is additionally token-identical to a cold no-cache prefill.

The tensor comparison is the load-bearing one. Sampled tokens are a weak probe: thirty-two greedy samples taken after the state is already fixed can miss a divergence a tensor diff cannot. An earlier draft gated only on token identity; that gate was replaced.

The positive control confirms the comparator is sensitive — caches over differing content diverge by up to 423 in maximum absolute difference. Against that baseline, one incidental finding stands out: prefilling in 256-token versus 512-token chunks produces bit-identical state, zero difference across all 60 recurrent and 20 attention tensors on the 35B. The gated-delta recurrent scan is invariant to chunk boundary. The quantity we were most concerned about perturbing does not perturb.

Cost of a single persist

The serialiser holds a process-global evaluation lock for the entire write. Not merely the model container — every tensor evaluation anywhere in the process blocks, including work belonging to a different model.

07Claims withdrawn, and then reinstated

Three claims from an earlier draft were falsified by the instrument built to check them, and duly withdrawn. Measuring at production scale then reinstated all three. The sequence is worth recording in full, because the intermediate position was wrong in a way that reads as rigour.

Round trip — withdrawn at 16k, reinstated at 183k

The estimate claimed roughly 17× less round-trip I/O, counting eliminated reads as recovered cost. At 16,384 tokens device reads measured near zero: nothing in the serialisation layer flushes to disk, so a snapshot written seconds earlier is still resident and its re-read is served from RAM. The claim was cut to the write half alone.

At 183,040 tokens device reads were 11.90 GB. A 3.8 GB snapshot cannot stay resident across a resume alongside 20 GB of model weights, so the re-read becomes real disk traffic. The read half is not free; it is free only below the size where it matters.

Wall clock — withdrawn at 16k, reinstated at 183k

Overhead was estimated at 22% of warm time by subtracting an assumed prefill rate from observed round times. Direct measurement of both arms at 16k gave 1.02× — the residency arm's per-round time rises on the same curve as the baseline while performing no I/O at all, so that growth was never I/O cost. It is attention cost scaling with context.

At production length the honest measurement is 1.25×, 111 seconds saved per document, with I/O accounting for 19.9% of warm time. The original estimate was right by accident and the correction was right for the wrong workload.

Persist stall — withdrawn at 16k, reinstated at 183k

A process-wide freeze of 0.6–2.0 seconds was projected from assumed disk bandwidth, then withdrawn when isolated measurement showed 45–281 ms. The write reaches the page cache at roughly 6 GB/s and returns; it never touches the device synchronously.

At production snapshot size a single save holds the process-global evaluation lock for 8.54 seconds — worse than the projection it replaced.

The common cause is that every one of these quantities scales with snapshot size while prefill time scales with token count, and the two grow at different rates. A benchmark an eighth of the production size does not understate the effect proportionally; it inverts the conclusion. The corrections were not sloppiness in the first draft or diligence in the second — both were artifacts of measuring at a convenient size.

Separately, four benchmark gates failed on first run and were traced to defects in the harness rather than the subject: an off-by-one in the largest calibration point; a control that perturbed a genuine invariance instead of testing instrument sensitivity; a cache-tier probe that sampled one call too early; and a state comparison that ran after generating from the cache it was comparing, so it diffed 16,174 tokens against 15,360. A fifth failure was real: warm released a completed cache whether or not the write had happened, so under a no-persist policy it finished, wrote nothing, and silently discarded the entire warm. That one shipped as a bug and was caught by a gate, which is the argument for the gates.

08What is not claimed

  • The wall-clock gain is workload-dependent, not universal. It is 1.25× at 183k tokens and 1.02× at 16k on the same model. Quote the size with the number.
  • Timings are single observations. One run per cell; the 9B ratio moved 1.01×–1.03× across three whole-harness runs. Byte counts are deterministic and do not carry this caveat.
  • All measurements use an unquantized bf16 KV cache with an effectively unbounded disk budget and the RAM tier disabled, which isolates the warm loop but means the eviction and quantization paths are unexercised. The byte model's per-element factor assumes bf16 and would need changing for a quantized cache.
  • Apple silicon, unified memory, one host. The page-cache behaviour that dominates the 16k results is a property of a 137 GB machine.
  • Residency's memory cost is bounded but not free. Peak memory during a warm is unchanged — the baseline reloads the same cache it would have held — but a held warm keeps roughly 20 KiB per token resident between resumes, where the baseline keeps nothing. That is a cost only under concurrency or long idle gaps.

09Assessment

The reported problem was volume — tens of gigabytes written to warm a single document. At the production workload it is 38.60 GB reduced to 3.81 GB, with 34.79 GB of reads eliminated outright, and a cache held across pause and resume that is bit-identical to one built in a single pass.

The effect on felt latency is real but smaller and size-dependent: 1.25× on the warm itself, and seventeen eight-second freezes of every tensor evaluation in the process reduced to one. Below roughly 32k tokens none of this is worth doing, which is the most useful thing the measurements say.

Two results transfer beyond this package. The cost model is exact and architectural: for any future model, both constants follow from its configuration file before it is ever loaded, which turns cache sizing from a measurement exercise into arithmetic.

And the defect was not a typo. [KVCache] has no Sendable conformance and the model container's perform is generic over a Sendable return, so a resumable background prefill cannot return its live cache out of the serialised domain — the filesystem becomes the only channel available between resumes. Combined with a catalog that re-plans every boundary once a single block is new, and a writer with no delta path, quadratic write amplification is the shape the type system selects for. The fix is a typed residency holder, not a faster writer.
